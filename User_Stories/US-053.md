# US-053 · System — Manual Entry Validation & Rate Limiting
**Epic:** 13 — Sequential QR Numbering  
**Priority:** P1  
**Status:** Open  
**Phase:** 3 (Backend — Manual Redemption Endpoint)

---

## User Story
> As a **System**, I want to validate manual QR number entries and apply rate limiting, so that the manual entry fallback cannot be abused to enumerate or brute-force valid QR codes.

---

## Background
Sequential numbers (1, 2, 3...) are predictable — unlike UUIDs. Without protection, a bad actor could enumerate all org QR codes. Rate limiting and audit logging are mandatory for this feature.

---

## Acceptance Criteria
- [ ] Input validation: `qr_number` must be a positive integer
- [ ] Input validation: `org_code` must be a valid, active organisation
- [ ] Rate limiting: max 5 failed lookups per IP address per hour
- [ ] On 6th failed attempt: return HTTP 429 _"Too many attempts. Try again in X minutes."_
- [ ] Successful lookup resets the failure count for that IP
- [ ] All attempts (success and failure) are written to an `audit_log` table: `{ ip, org_code, qr_number, result, timestamp }`
- [ ] Admin can view manual entry audit log in the admin panel (read-only)

---

## Rate Limit Implementation
Use in-memory store (or Redis if available) keyed by `ip:org_code`:
```
key: manual_entry:192.168.1.1:TESTORG
value: { count: 3, window_start: 1710000000 }
TTL: 3600 seconds
```

---

## Error Responses
| Condition | HTTP | Message |
|-----------|------|---------|
| Invalid org_code | 400 | "Organisation not found" |
| qr_number not a number | 400 | "Invalid QR number format" |
| QR not found | 404 | "QR number not found for this organisation" |
| QR found but invalid state | 409 | (same messages as US-018 to US-022) |
| Rate limit exceeded | 429 | "Too many attempts. Try again later." |
