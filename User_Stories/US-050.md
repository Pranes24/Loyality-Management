# US-050 · System — Sequential QR Numbers (Org-Wide)
**Epic:** 13 — Sequential QR Numbering  
**Priority:** P1  
**Status:** Open  
**Phase:** 1 (DB Migration) + Phase 2 (Backend — Number Assignment)

---

## User Story
> As a **System**, I want to assign each QR code a unique sequential number within its organisation, so that every QR sticker carries a human-readable identifier that can be used for manual entry or tracking.

---

## Background
Each QR code has two identifiers:
1. **UUID** (`id`) — encoded in the actual QR image; random, non-guessable; used for scanning
2. **Sequential number** (`qr_number`) — printed on the sticker label; integers starting at 1 per org; used for manual entry

Numbering is **org-scoped and permanent**:
- Org A: 1, 2, 3 … 50,000
- Org B: 1, 2, 3 … 30,000
- Numbers are assigned in order as batches are created
- Once assigned, a number never changes or is reused

---

## Acceptance Criteria
- [ ] `qr_codes` table has a `qr_number` column (integer, NOT NULL)
- [ ] `qr_number` is unique within an org: `UNIQUE(org_id, qr_number)`
- [ ] Numbers are assigned sequentially starting from 1 for each org
- [ ] When a new batch is created with N codes, numbers assigned are: `(current_max + 1)` to `(current_max + N)`
- [ ] Assignment is atomic — concurrent batch creation cannot produce duplicate or skipped numbers
- [ ] `qr_number` is included in:
  - QR export ZIP (CSV/metadata)
  - Batch detail QR table
  - Admin user detail scan history

---

## Data Model
```sql
ALTER TABLE qr_codes ADD COLUMN qr_number INTEGER NOT NULL DEFAULT 0;
ALTER TABLE qr_codes ADD CONSTRAINT qr_codes_org_number_unique UNIQUE (org_id, qr_number);
```

---

## Number Assignment Logic (Batch Service)
```sql
-- Get next starting number (atomic, within transaction)
SELECT COALESCE(MAX(qr_number), 0) + 1 AS next_num
FROM qr_codes
WHERE org_id = $orgId
FOR UPDATE;

-- Then insert QR codes with numbers next_num, next_num+1, next_num+2 ...
```

---

## Edge Cases
- First batch for org: starts at 1
- Deleted QR codes: numbers are NOT reused (gaps are acceptable)
- Existing QR codes (before this migration): backfilled with sequential numbers in created_at order
