# US-049 · Batch Creation — Quota Validation
**Epic:** 12 — QR Quota Management  
**Priority:** P1  
**Status:** Open  
**Phase:** 2 (Backend — Quota Check in Batch Creation)

---

## User Story
> As an **Org Admin**, I want the system to prevent me from creating a batch larger than my remaining QR quota, so that I cannot accidentally exceed my allocated limit.

---

## Background
When creating a batch the org admin specifies the number of QR codes to generate. Before generating, the system must check: `requested_count <= (qr_quota - qr_used)`. If the check fails, the batch is not created and a clear error is returned.

---

## Acceptance Criteria
- [ ] Backend validates `requested_count <= qr_remaining` before creating any QR codes
- [ ] If validation fails, return HTTP 400 with message: _"Insufficient quota. Requested: X, Remaining: Y"_
- [ ] If quota = 0, return HTTP 400: _"No QR quota assigned to this organisation"_
- [ ] Frontend shows the error inline on the batch creation form — does not navigate away
- [ ] The batch is NOT partially created if quota is insufficient — all-or-nothing
- [ ] After successful batch creation, quota usage updates immediately in the UI
- [ ] Remaining quota is shown live on the form as the user types the QR count

---

## API
| Method | Endpoint | Validation Added |
|--------|----------|-----------------|
| POST | `/api/batch` | Check `count <= qr_remaining` before INSERT |

---

## Error Response
```json
{
  "error": "Insufficient quota. Requested: 5000, Remaining: 2400"
}
```

---

## Edge Cases
- Concurrent batch creation: use a DB-level check (SELECT ... FOR UPDATE on org row) to prevent race conditions
- Exact quota: allowed — org admin can use every single code in their quota
